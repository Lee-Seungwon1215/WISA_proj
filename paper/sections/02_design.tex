\section{Framework Design}
\label{sec:design}

CT-KAT (invoked as \texttt{ctkat}) is a fail-closed CT screening framework for
PQC. It binds three checkers under one declarative configuration, sweeps each
harness across a compiler${}\times{}$optimization matrix, and routes findings
through a default-deny taxonomy backed by a cited accepted-variable-time registry.
The goal is not to prove leak absence, but to make screening honest: report the
build context, expose each checker, and never silently promote a CT failure to a
key-recovery leak.

\subsection{Unified Configuration and Pipeline}
\label{ssec:config}

One YAML file describes the harnesses, secret regions, build configurations, and
per-layer options. It keeps build flags, tainted regions, and timing parameters
in one source of truth. Each harness is compiled per build cell, screened by the
three layers, and reduced by triage to one \texttt{verdict\_class}.

\subsection{Three Checker Layers}
\label{ssec:checkers}

CT side channels share no single observable signature, so CT-KAT composes three
checkers with different blind spots.

\paragraph{Layer 1: structural CT check (Valgrind/Memcheck).}
Following ctgrind~\cite{ctgrind} on Memcheck~\cite{valgrind}, secret inputs are
marked undefined and tainted branches or addresses are reported. This checks
program structure, not instruction latency.

\paragraph{Layer 2: variable-latency instruction scan (asm-scan).}
\texttt{asm-scan} is a warn-only scan over emitted assembly, across optimization
levels, for candidate variable-latency instructions such as integer division and
remainder. It catches the KyberSlash class~\cite{kyberslash}, but remains
warn-only because the operand may be public; triage resolves public versus secret.

\paragraph{Layer 3: black-box statistical timing (dudect).}
A black-box test after dudect~\cite{dudect} runs two input classes and applies a
Welch $t$-test to cycle counts. It needs no causal model, so it catches timing
dependence with no structural branch and no recognizable variable-latency
instruction, but it is environment-sensitive and reports noise explicitly.

\subsection{Build-Configuration Sweep (ct-matrix)}
\label{ssec:matrix}

A constant-time property belongs to the compiled program. \texttt{ct-matrix}
recompiles each harness under named compiler/cflags cells and re-runs the same
structural check. Our image uses gcc~13.3.0 and clang~18.1.3 over debug
(\texttt{-O0} with debug/no-inline flags), release (\texttt{-O2} with debug
metadata and no LTO), and size (\texttt{-Os}) builds. The sweep matters because
optimization can remove a debug branch or inline leak-site functions into a
parent frame; CT-KAT reports the build-dependence and unions leak-site functions
for triage.

\subsection{Default-Deny Triage Taxonomy}
\label{ssec:triage}

Over-claiming every structural failure as key recovery is dangerous because PQC
contains accepted variable-time behavior, such as ML-DSA rejection
sampling~\cite{fips204,dilithium}. CT-KAT assigns each harness one
\texttt{verdict\_class}:
\begin{description}
  \item[\texttt{robust}:] all checkers pass across the matrix.
  \item[\texttt{ct-clean-untriaged}:] structural pass; another layer's candidate
        (e.g.\ a division) is untriaged.
  \item[\texttt{varlat-secret-risk}:] structural pass plus a
        \emph{secret-derived} variable-latency instruction (KyberSlash).
  \item[\texttt{build-sensitive-ct}:] the structural verdict is unstable across
        the matrix.
  \item[\texttt{accepted-variable-time}:] structural fail, but \emph{every}
        leak-site function is a registered, cited design behavior.
  \item[\texttt{needs-analysis}:] structural fail with some unregistered
        leak-site function; a human must rule.
  \item[\texttt{ct-leak}:] a confirmed leak, reachable \emph{only} by explicit
        human verdict; never assigned automatically.
\end{description}

The \texttt{accepted-variable-time}/\texttt{needs-analysis} boundary is a cited
registry (\texttt{docs/accepted\_variable\_time.md}). If all leak-site functions
are registered the harness is accepted; if any is unrecognized it stays
\texttt{needs-analysis}---never silently accepted, never silently promoted to a
leak. Section~\ref{sec:results-coverage} shows this on ML-DSA-65.
