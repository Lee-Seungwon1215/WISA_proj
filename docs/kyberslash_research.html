CT-KAT × KyberSlash — 기능 추가 검토

CT-KAT이 지금 못 보는 누수 한 종류(비밀이 나눗셈/곱셈 같은 가변시간 연산에 들어가는 것)를 짚고, 그 탐지를 이 코드에 기능으로 넣으면 얼마나 좋아지나 / 어떻게 넣나를 정리한 메모. KyberSlash는 그 누수의 대표 예시. 논문/신규성 얘기 아님 — 그냥 도구 완성도용 기능 검토.

0. 한눈에 (TL;DR) #

  • CT-KAT은 지금 비밀 분기와 비밀 메모리 접근은 잡지만, 비밀이 나눗셈/곱셈(가변시간 연산)에 들어가는 누수는 못 본다.
  • KyberSlash가 그 대표 사례 — ML-KEM(Kyber)의 secret / KYBER_Q 나눗셈이 일부 플랫폼에서 가변시간이라 비밀이 샌다.
  • 그래서 지금 코드로 취약 Kyber를 돌리면 → CLEAN(놓침).
  • 이 “연산자(operand) 타이밍” 탐지를 CT-KAT에 4번째 검사로 넣으면 분기·메모리·연산자 3축을 다 커버해서 도구가 훨씬 완성도 높아진다.
  • 솔직 이걸 새 연구로 팔기는 약하다. Binsec/Rel·KyberSlash 논문의 패치드 Valgrind·saferewrite·FaCT 등 이미 여러 도구가 하는 “operand model”이다. 여기선 그냥 CT-KAT을 더 쓸모있게 만드는 기능으로 본다 (§4.6).

1. 이 프로그램(CT-KAT)이 뭐냐 #

C/C++ 암호 구현을 던지면 constant-time 위반 = 비밀이 실행시간으로 새는지를 자동 검사.

build → KAT(정확성) → ct(Valgrind) → dudect(통계 타이밍) → 통합 verdict

1.1 세 개의 검사 레이어

레이어무엇을 보나코드
ct (Valgrind/Memcheck)비밀이 분기 조건 또는 메모리 주소에 쓰이나 (구조적)valgrind_parser.py
dudect비밀 fixed-vs-random으로 실행시간 분포가 다른가 (Welch t-test)statistics.py, dudect_runner.py
verdict둘을 합쳐 CLEAN / STRUCTURAL_LEAK / SUSPECT / RISKY / CRITICAL / INCONCLUSIVEverdict.py

1.2 ct 레이어가 “잡는 것”의 정확한 목록

valgrind_parser.py의 _FINDING_CLASSIFIERS가 잡는 패턴:

  • "Conditional jump or move" → SECRET_DEPENDENT_BRANCH (비밀로 분기)
  • "Use of uninitialised value" → SECRET_DEPENDENT_VALUE_USE (비밀 값 사용)
  • "Invalid read/write" 등 → MEMORY_ERROR
  • 함수 이름이 sbox/ttable/lookup/_table 등이면 VALUE_USE → SECRET_DEPENDENT_MEMORY_ACCESS 승격
여기가 빈칸: 이 목록 어디에도 “비밀이 나눗셈/곱셈 명령어에 들어감”은 없다. Memcheck는 비밀(undefined로 마킹)이 분기 / 주소 / 시스템콜에 도달할 때만 에러를 낸다. 비밀이 div에 들어가 나오는 결과는 그냥 “undefined 전파”라 에러가 안 난다. → 여기에 기능을 넣자는 게 이 문서.

1.3 dudect 레이어의 측정 축

timing_kem.c.j2의 leak_target: sk(비밀키, 기본) · ct(암호문) · fo(유효 ct vs 무효 ct = FO fallback / implicit-rejection 경로).

1.4 이미 알고 있는 한계 (코드/문서에 박혀 있음)

  • F9 (cflags 비대칭): ct는 -O0, dudect는 -O2 → 같은 소스라도 다른 binary를 분석 → _print_cflags_banner 경고. KyberSlash와 직결(§4).
  • report.py 권고문: VALUE_USE에 “Verify the operation has data-independent latency on target CPU (esp. div/mul).” ← div/mul 사각지대를 알면서 탐지는 안 함 (= 이 기능을 넣을 자리라는 힌트).
  • README.md: KyberSlash 명시적 면책 — 단 표현은 살짝 부정확(§3.4).

2. KyberSlash가 뭐냐 #

ML-KEM(Kyber)의 secret-dependent 나눗셈 타이밍 누수. 두 변종:

이름위치(함수)단계
KyberSlash1poly_tomsg복호화/decap에서 비밀 다항식 → 메시지 디코딩
KyberSlash2poly_compress / polyvec_compress(재)암호화 압축

문제의 코드는 본질적으로 어떤_secret값 / KYBER_Q (KYBER_Q = 3329) 꼴. 이 나눗셈이 피연산자에 따라 시간이 달라지는 기계어로 컴파일되면 비밀이 샌다.

2.1 이 repo 안의 실제 코드 — 취약 vs 수정 구체

이 저장소(examples/pqc_mlkem768/clean/poly.c)는 이미 수정된 PQClean 버전이고, 취약했던 원래 코드가 주석으로 그대로 남아 있다.

poly_tomsg (= KyberSlash1):

t = a->coeffs[8 * i + j];
// t += ((int16_t)t >> 15) & KYBER_Q;
// t  = (((t << 1) + KYBER_Q/2)/KYBER_Q) & 1;   // ← 취약 (원래 코드): /KYBER_Q
t <<= 1;
t += 1665;        // 1665 = round(KYBER_Q/2)
t *= 80635;       // 80635/2^28 ≈ 1/3329
t >>= 28;         // → (t * 80635) >> 28  ≈  t / 3329  (상수시간 곱셈+시프트)
t &= 1;

poly_compress (= KyberSlash2)도 동일 패턴: 원래 /KYBER_Q 한 줄이 주석으로 남고 d0 *= 80635; d0 >>= 28; 로 교체돼 있다.

2.2 수정의 원리 — reciprocal-multiply (“나눗셈을 곱셈으로”)

x / 3329 를 직접 나누지 않고 (x * 80635) >> 28 로 계산한다.

  • 80635 / 2^28 = 0.000300388… ≈ 1/3329 = 0.000300390… → 곱하고 28비트 내리면 3329로 나눈 것과 같음.
  • 이건 컴파일러가 -O3에서 자동으로 하는 “magic number division”을 소스에 손으로 박은 것. 소스에 박았으니 컴파일러/플래그/아키와 무관하게 상수시간 곱셈+시프트 (= 안전).
  • 참고: 나눗셈→곱셈으로 바꿔도 일부 CPU는 곱셈조차 가변시간이라, 엄밀히는 saferewrite가 div·mul 둘 다 경고 (실무상 32비트 mul은 거의 항상 상수시간).

2.3 실제 익스플로잇 사실관계

  • 시연 플랫폼: Raspberry Pi 2 (Arm Cortex-A7), Arm Cortex-M4.
  • 비밀키 복구: KyberSlash2는 수 분, KyberSlash1은 수 시간.
  • KyberSlash2는 원격(공격자가 ct 보내고 응답시간 측정)으로도 일부 가능.
  • 영향: Kyber 공식 reference 코드 포함 다수 구현. 수정 = 위 reciprocal-mul.

2.4 “어디서 취약하냐”는 플랫폼·컴파일러에 따라 다르다 중요

같은 C 소스라도 컴파일 결과(기계어)가 달라 취약점이 있기도/없기도 함 (원래 /KYBER_Q 코드 기준 — repo의 fix는 이미 곱셈이라 안전):

환경secret / KYBER_Q 의 기계어취약?
x86 gcc -O3 (속도)곱셈(reciprocal trick)대체로 안전
x86 gcc -Os (크기)division 명령어취약 (Zen2: 분자 8192↑ 시 +1 cycle)
RISC-V -Os/-O3둘 다 division취약
ARM Cortex-M/A -Os__divsi3 소프트나눗셈 → 입력 의존 분기취약

3. 지금 CT-KAT의 사각지대 (왜 이 기능이 필요한가) #

3.1 지금 코드로 취약 Kyber를 돌리면? → 못 잡는다

  1. ct (Valgrind): vanilla Memcheck는 분기/주소만 봄 → §1.2 classifier에 div 항목 없음.
  2. dudect: 잡으려면 binary에 진짜 가변시간 div가 있어야 하는데 —
    • 디폴트가 x86 -O2 → 컴파일러가 div를 곱셈으로 교체 → 시간차 없음.
    • -Os 강제해도 x86 div는 미세 + 디폴트가 QEMU(Apple Silicon Docker) → rdtsc 불신 → monotonic 강등 → 노이즈에 묻힘.
    • 진짜 신호는 ARM/RISC-V 실하드웨어인데 거긴 안 봄.

예상 출력 (추정 — valgrind 미설치 환경이라 실측 아님):

[CTKAT] Build: PASS
[CTKAT] KAT: PASS
[CTKAT] Constant-Time Check: PASS   ← div를 안 봄
[CTKAT] dudect Timing Check: PASS   ← x86/-O2라 시간차 없음
→ verdict: CLEAN                     ← 취약본인데 초록불 (놓침)

3.2 시험용 취약본 만드는 법 (기능 테스트용) 구체

기능을 넣으면 “정말 잡나”를 시험해야 한다. §2.1의 주석 한 줄만 되살리면 취약 버전이 된다:

  1. poly.c의 poly_tomsg에서 t <<= 1; t += 1665; t *= 80635; t >>= 28;를 지우고 주석 원본 t = (((t << 1) + KYBER_Q/2)/KYBER_Q) & 1;로 교체 (poly_compress도 동일).
  2. CT-KAT 통합 테스트용이면 ct 기본 -O0로 컴파일한 뒤 objdump에서 div/idiv/udiv/sdiv가 실제로 나오는지 확인한다. 배포 취약성 재현용이면 gcc -Os, RISC-V/ARM 등 KyberSlash 논문이 문제 삼은 조합을 따로 본다.
  3. 이러면 “취약 코드 + DIV가 실제로 남은 binary” = 시험용 양성 샘플. 고친 버전은 그대로 음성 샘플.
참고: KAT(정확성)는 두 버전 다 PASS여야 정상 (수학적으로 같은 결과). 차이는 오직 시간/명령어에만 있음 → 새 기능이 봐야 할 바로 그 축.

3.3 탐지 커버리지 매트릭스 (누가 뭘 잡나)

누수 종류예시ct (Valgrind)dudect새 기능(§4)
비밀 분기if (secret)✅△—
비밀 메모리 인덱싱sbox[secret]✅△—
비밀 나눗셈/곱셈 (KyberSlash)secret / Q❌❌(x86/-O2)·△(ARM)✅

△ = 플랫폼/최적화/노이즈에 따라 잡힐 수도 안 잡힐 수도. 빈칸 = 이미 다른 레이어가 커버. 맨 아랫줄 빈칸을 메우는 게 이 기능.

3.4 README 면책 표현 정정거리 (덤)

KyberSlash를 “adversarial-input / 희귀 ct 트리거 류”로만 묶으면 부정확하다. KyberSlash1은 정상 decap 중 비밀 디코딩에서 새는 거고, KyberSlash2도 정상 재암호화/압축 경로의 operand-timing 문제다. ct-leak 모드의 랜덤샘플 한계와도 관련은 있지만, 본질은 산술 명령어 latency 쪽이다. README와 템플릿 주석은 v3에서 이 둘을 분리해 고쳤다.

4. 기능을 어떻게 넣나 — “어셈블리 등” #

4.0 “어셈블리”가 무슨 뜻이었나

어셈블리로 하니스를 짜라는 게 절대 아니다. 두 의미가 섞여 있음:

  1. 취약점 존재 여부가 “생성된 어셈블리”에 달려 있다 — 같은 C라도 컴파일러가 div를 내면 취약, mul로 바꾸면 (대체로) 안전 → 컴파일된 기계어를 열어 div가 있는지 확인(Godbolt/objdump).
  2. 탐지를 명령어 수준에서 해야 한다 — 소스/분기 수준이 아니라 어떤 기계어에 비밀이 들어가나를 봐야 함.

4.1 같은 /3329가 -O에 따라 어떻게 달라지나

// 소스
uint16_t t = secret_coeff;
r = ((t << 1) + 1664) / 3329;

x86-64 gcc -O3  →  imul + shr   (곱셈+시프트, 대체로 안전)    ← 보통 안전
x86-64 gcc -Os  →  div          (나눗셈 명령, 피연산자 의존)  ← 취약
Apple clang -O0/-Oz(x86/arm64) → div/udiv/sdiv 가능          ← ct 단계에서 보임
riscv  gcc -O*  →  divu         (나눗셈)                     ← 취약 가능

확인: objdump -d <binary> | grep -E 'div|idiv|udiv|divu' 또는 Godbolt. → CT-KAT의 F9(ct=-O0/dudect=-O2)가 바로 이 축이다.

4.2 구현 방법 3가지 (난이도/정확도 순)

(A) 정적 스캔 — objdump/Capstone 디스어셈 PoC 시작점

  • 컴파일된 하니스 디스어셈 → div/idiv(x86), udiv/sdiv(ARM), div/divu(RISC-V) 찾음.
  • 그 명령의 피연산자가 비밀에서 유래했는지 역추적 (use-def, 또는 보수적으로 “비밀 버퍼 만지는 함수 안에 div 있나”).
  • 장점: gcc만 있으면 됨, x86/QEMU에서도 동작(타이밍 불필요), 끼우기 쉬움.
  • 단점: 정밀 taint 아님 → false +/− 존재. PoC·1차 스크리닝엔 충분.

(B) Valgrind/VEX 패치 — Memcheck 확장 정밀 최종형

  • KyberSlash 저자가 한 방식. VEX IR로 올릴 때 tainted shadow가 Iop_DivU*/DivS* 등 나눗셈 op에 도달하면 경고.
  • 장점: CT-KAT가 이미 Valgrind 래핑 → 일관, 정밀 taint(이미 VALGRIND_MAKE_MEM_UNDEFINED로 비밀 마킹).
  • 단점: 패치드 valgrind 빌드·Docker 반영 필요, 난이도 높음.

(C) DBI — Pin/DynamoRIO 명령어 수준 taint

  • 정밀하나 인프라 무겁고 CT-KAT 스택과 이질적. 우선순위 낮음.

4.3 비밀 “마킹”은 이미 돼 있다 (재활용)

kem/sign 하니스는 이미 VALGRIND_MAKE_MEM_UNDEFINED(sk + offset, len)으로 무엇이 비밀인지 valgrind에 알려준다(harness_kem.c.j2). (B)는 이 shadow를 div-체크에 재사용 → 별도 taint 불필요. (A)는 이 offset/length(R-2에서 검증됨)를 비밀 범위 힌트로 써서 역추적 시드로 활용.

4.4 asm_scan 실제 구현 (v8 — 멀티-opt, warn-only)

v8 업데이트: 아래는 처음 구상한 흐름과 다르다. Phase 0 실측(docs/kyberslash_direction.md §8.7)에서 gcc는 상수 /KYBER_Q를 -O0에서도 역수곱으로 죽인다는 게 드러났다. 그래서 "ct(-O0) 한 빌드만 보고 Finding으로 verdict에 합류"하는 옛 방식은 폐기했다(그 빌드엔 div가 아예 없어 0개 탐지). 실제 구현은 여러 -O로 컴파일해 어느 빌드에서 div가 살아남나를 보고, 결과는 verdict와 분리된 warn-only artifact로 낸다.

ctkat asm-scan -c ctkat.yaml   (독립 서브커맨드):
  1. ct.harnesses[].sources 각각을 -O0/-Os/-O2(+ct의 실제 -O)로 -c 컴파일
  2. objdump -d → div/idiv/udiv/sdiv/divu/rem 위치 수집 (mul/FP-div 제외)
  3. nm 심볼테이블로 함수명 보정 (Mach-O ltmp0 → 실제 함수명)
  4. (source, function)별로 "어느 -O에서 div가 살아남나" 집계
  5. ctkat_varlat_candidates.csv/json 으로 출력 — verdict는 절대 안 건드림

taint 없음: 비밀 여부를 증명하지 않고 sources 안의 모든 division을 후보로 낸다 (그래서 fips202의 public division도 뜬다). 그래서 warn-only다 — 정밀 taint는 patched Valgrind(방법 B, 아직 보류)의 몫이다. FindingType에 합류시키면 "finding 있으면 FAIL" 규칙 때문에 오탐이 CI를 깨므로, 의도적으로 별도 artifact로 분리했다.

4.5 컴파일러/최적화 매트릭스 (덤)

CT-KAT는 이미 dudect.compiler.cflags/ct.cflags/shared_cflags로 플래그 제어 + qemu_detect/clock=auto로 타이밍 환경을 인지한다. 단 이건 cross-toolchain 매트릭스가 아니다. 같은 소스를 여러 {compiler, -O, target}로 컴파일해서 “어디서 div가 박히나” 비교하려면 별도 cross-compiler/실하드웨어 또는 QEMU 실행 매트릭스가 필요하다.

4.6 솔직: 논문 novelty로 팔기 약하다 정직

이 “비밀→div/mul 탐지”는 CT 분야에서 이미 알려진 누수 모델(operand / variable-time instruction model)이고, 여러 도구가 이미 한다. CT-KAT에 넣는 건 새 원리 발명이 아니라 이 프로젝트에 빠진 축을 통합하는 일에 가깝다.

부록 A. 용어 사전 #

  • constant-time (상수시간): 실행시간이 비밀값에 의존하지 않는 성질. 위반하면 타이밍으로 비밀 추론 가능.
  • operand / variable-time instruction model: 비밀이 가변시간 명령어(div/mul 등)에 들어가는 걸 누수로 보는 모델. 분기·메모리 모델에 이은 세 번째 표준 축.
  • ctgrind / TIMECOP: Valgrind/Memcheck로 비밀을 “uninitialized”로 마킹해 분기/주소에 쓰이면 잡는 상수시간 검사.
  • dudect: fixed-vs-random 입력으로 실행시간 분포를 Welch t-test로 비교하는 통계적 검사 (Reparaz et al. 2017).
  • taint / shadow: “이 값은 비밀에서 유래” 추적 메타데이터. Memcheck의 undefined-bit shadow가 사실상 taint.
  • reciprocal-multiply (magic number division): x / C를 (x * M) >> k로 바꾸는 상수시간 나눗셈. M/2^k ≈ 1/C.
  • VEX IR: Valgrind 내부 중간표현. 명령어를 IR op(Iop_DivU32 등)으로 올려 계측. div 탐지를 여기 추가.
  • FO / implicit rejection: KEM이 무효 ct를 받으면 타는 재암호화·거부 경로. CT-KAT의 leak_target: fo가 검사하는 축.
